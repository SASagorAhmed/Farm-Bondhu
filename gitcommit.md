# Git Commit Policy

## Repo hooks (strip injected trailers)

Some environments append lines like `Made-with: Cursor` to `git commit` via `--trailer`. This repo includes a `commit-msg` hook that **removes** those lines so messages stay professional.

One-time setup after clone (run from repo root):

```bash
git config core.hooksPath .githooks
```

On Windows (Git Bash or any shell in the repo root), the same command applies. Hooks in `.githooks/` are versioned; `core.hooksPath` is local to your machine and is not committed.

## Core Rules

1. Commit messages must be human and professional.
2. Never include AI/tool attribution lines (for example: `Made-with: Cursor`, `Generated by AI`).
3. Use clear commit types:
   - `feat(scope): ...`
   - `fix(scope): ...`
   - `docs(scope): ...`
   - `refactor(scope): ...`
   - `chore(scope): ...`
4. Write what changed and why in simple, direct language.
5. Keep subject line concise (recommended <= 72 chars).

## Workflow Rule (Very Important)

Always work and commit part by part, not all at once.

- Make a small logical change.
- Commit that small part.
- Push that commit.
- Continue with next small part.
- Commit again.
- Push again.

In short: commit-push, then commit-push, continuously by logical parts.

## Allowed Examples

- `feat(admin): add approval queue filters for pending requests`
- `fix(auth): handle expired token during profile fetch`
- `docs(deploy): add vercel deployment checklist`

## Forbidden Examples

- `update files`
- `final work`
- any commit body containing:
  - `Made-with: Cursor`
  - `Generated by AI`
  - similar tool attribution text

## Commit Template

Subject:
`type(scope): short summary`

Body (optional):
- What changed
- Why it changed
- Notes for reviewer/tester
